# Phase 2 Review Checklist — Spring Boot 확장 점검표

## 0) 학습 범위 인식
- [ ] 현재 역량과 다음 단계 목표를 문서로 구분했는가?

## 1) 유스케이스 정확성
- [ ] 주문 생성 성공/실패 흐름이 명확한가?
- [ ] 상태 전이(PENDING/PAID/PAYMENT_FAILED)가 누락 없는가?
- [ ] 중복 요청 시 idempotency가 보장되는가?

## 2) 설계/트랜잭션
- [ ] Controller/Application/Domain/Infra 경계가 분명한가?
- [ ] 트랜잭션 범위가 과도하지 않은가?
- [ ] 외부 결제 호출 장애 시 일관성 전략이 있는가?
- [ ] ADR에 대안 비교와 선택 이유가 있는가?

## 3) JPA/성능
- [ ] N+1 개선 전/후 SQL 수치가 있는가?
- [ ] 페이징 전략 선택 근거가 있는가?
- [ ] fetch join 사용 시 페이징 이슈를 인지/회피했는가?
- [ ] p95 개선 결과가 수치로 제시되는가?

## 4) 예외/표준
- [ ] 공통 에러 포맷이 전 API에 일관 적용되는가?
- [ ] 비즈니스/시스템 예외 분리가 되어 있는가?
- [ ] traceId가 오류 응답/로그 모두에 포함되는가?

## 5) 테스트
- [ ] 테스트 총 40개 이상인가?
- [ ] 단위/통합/API 분리가 되어 있는가?
- [ ] 실패/경계 케이스가 충분한가?
- [ ] 회귀 방지 테스트가 존재하는가?

## 6) 운영성
- [ ] 로그 필드(traceId/orderId/userId/status)가 표준화됐는가?
- [ ] 메트릭(성공률/실패율/p95/재시도)을 수집하는가?
- [ ] 장애 대응 기준(runbook/알람 조건)이 정리됐는가?

## 7) 회고
- [ ] 기술 부채 3개를 식별했는가?
- [ ] 다음 4주 개선 계획이 현실적인가?
- [ ] 팀 공유 가능한 학습 요약이 있는가?
